# 工程基础

## 设计

### 职责

工程基础模块负责统一项目运行环境、依赖版本、编译规则、代码检查、测试入口和 Cloudflare Pages 静态输出。它不包含文章内容模型和具体页面功能，这些能力会在后续模块中实现。

### 目录结构

| 路径                            | 用途                                          |
| ------------------------------- | --------------------------------------------- |
| `app/`                          | Next.js App Router 根布局、基础页面和全局样式 |
| `app/icon.jpg`                  | 复用原型头像作为站点图标                      |
| `lib/site-config.ts`            | 站点基础信息、语言列表和站点地址校验          |
| `tests/unit/`                   | Vitest 单元测试                               |
| `tests/e2e/`                    | Playwright 用户流程测试                       |
| `.github/workflows/quality.yml` | 主分支和 PR 的自动质量检查                    |
| `next.config.ts`                | 静态导出和图片策略                            |
| `wrangler.jsonc`                | Cloudflare Pages 项目配置                     |

### 关键决策

1. 使用 Next.js App Router 和 TypeScript 严格模式。`noUncheckedIndexedAccess` 与 `exactOptionalPropertyTypes` 额外约束数组访问和可选字段。
2. 使用 Next.js `output: "export"` 输出到 `out`，Cloudflare Pages 只托管静态文件。未来的在线接口放在独立 `functions/` 目录。
3. HeroUI 3 与 Tailwind CSS 4 作为组件和样式基础。HeroUI 3 自带 React Aria 行为，不需要全局 Provider。
4. 组件从 HeroUI 子路径直接导入，减少无关模块进入客户端构建。
5. 图片使用本地文件和固定尺寸。静态导出阶段关闭 Next.js 在线图片优化，避免依赖常驻图片服务。
6. 页面默认使用 Server Component。只有主题、菜单、搜索和弹窗等需要浏览器状态的区域使用 Client Component。
7. E2E 使用 `next build` 的 `out` 产物和临时静态文件服务器，不依赖 Next.js 开发服务器。

## 实现细节

### 构建流程

1. pnpm 根据 `pnpm-lock.yaml` 安装固定依赖。
2. Next.js 编译 App Router 页面，并在构建阶段生成全部路由。
3. `output: "export"` 将 HTML、CSS、JavaScript 和图片写入 `out`。
4. Cloudflare Pages 直接发布 `out`，无需 Node.js 常驻进程。

### 质量检查

- Prettier 检查代码和文档格式。
- ESLint 检查 Next.js、React、TypeScript 和可访问性相关规则。
- `next typegen` 先生成 App Router 路由类型，`tsc --noEmit` 随后执行严格类型检查。
- Vitest 验证 TypeScript 纯函数，Node Test Runner 保留原型逻辑测试。
- Playwright 启动临时静态预览并验证构建产物可以真实访问。本地优先使用已安装的 Chrome，CI 使用版本固定的 Playwright Chromium。

### 环境变量

`NEXT_PUBLIC_SITE_URL` 用于生成 canonical、RSS 和分享地址。缺省值只服务于本地构建，正式部署必须在 Cloudflare Pages 中配置真实域名。`resolveSiteUrl` 会拒绝 HTTP(S) 以外的协议，防止生成不可用的公开地址。

### 扩展方式

- 新增动态能力时，在 `functions/api/` 下按接口职责拆分，并为请求和响应建立共享类型。
- 新增内容模块时，只通过 `lib/content` 暴露整理后的只读数据，页面层不直接读取文件。
- 新增第三方脚本时使用延后加载，并保持基础阅读流程在脚本失败时可用。
