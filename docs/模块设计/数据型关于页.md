## 职责与边界

本模块负责关于页的个人资料、GitHub 贡献、AI Token 汇总、微信读书状态、技术雷达和数据来源说明。品牌、作者、头像、社交链接和外部服务地址继续由 `site.config.ts` 统一管理。

关于页只读取公开数据。TokenBoard 的费用、缓存细节和其他内部统计不会进入浏览器响应；Notion Token、微信读书 Cookie 等私密凭据也不属于此模块的前端配置。

## 结构与数据流

```text
GitHub contribution HTML ─┐
                          ├─ Pages Function /api/about-status ─ AboutActivity
TokenBoard public JSON ───┘

data/reading-status.json ─── ReadingSection
site.config.ts ────────────── Hero + TechnologyRadar
```

- `AboutPageView` 是服务端页面组合层，输出 Hero、阅读、技术雷达和数据来源。
- `AboutActivity` 是页面里唯一需要浏览器请求的 Client Component，访问同源 `/api/about-status`。
- `loadAboutActivityStatus` 并行读取 GitHub 与 TokenBoard。每个来源单独超时和降级，一个失败不会清空另一个结果。
- Pages Function 返回公开缓存头，Cloudflare 可缓存 30 分钟，并在上游短时故障时继续使用旧响应。

## 关键决策

### 页面结构

正式页面沿用 `prototype/about.html` 已确认的白底、开放式分区和细分隔线。用户确认后删除 Hero 与活动区之间的深色“最近关注”横条。页面顺序固定为个人资料、最近 365 天、GitHub、TokenBoard、微信读书、技术雷达和数据来源。

### GitHub 贡献日历

`lib/about-status/github.ts` 使用 `parse5` 解析 GitHub 公开贡献页的结构化 HTML，读取每个日期的 `data-level`，并从关联的 `tool-tip` 提取精确贡献数。解析结果按日期排序，再计算活跃天数和总贡献数。页面结构缺失或活跃格子没有精确数量时直接判定该来源不可用，避免显示猜测值。

### TokenBoard 公开统计

TokenBoard 读取公共资料 JSON。Zod 只接收总 Token、今日、本月、来源分布和常用模型，输出对象重新逐字段组装。上游即使附带费用、缓存 Token 或其他字段，博客接口也会将它们丢弃。

当前公开接口没有最近 30 天逐日趋势，因此正式页面展示真实的来源比例和常用模型，没有沿用原型中的示意柱状图。

### 微信读书

`data/reading-status.json` 是可生成、可校验的本地数据契约，字段包括年度分钟数、读完数量、活跃天数、更新时间和书架。当前没有可靠的公开接口或部署凭据，初始文件保持空值，页面显示稳定的等待同步状态。后续同步脚本只能写入这份文件，页面组件不直接保存或读取 Cookie。

### 技术雷达

技术名称和 Adopt、Trial、Assess 阶段统一配置在 `site.config.ts` 的 `profileStatus.technologyRadar`。SVG 只负责把配置映射到固定视觉位置，图例使用 HeroUI `Chip`，不在组件中重复维护技术清单。

## 验证方式

- `tests/unit/about-status.test.ts` 覆盖 GitHub 数量与等级解析、无贡献日、空页面、缺失 tooltip、TokenBoard 字段裁剪和单来源失败。
- `tests/unit/site-config.test.ts` 覆盖 GitHub 用户名、TokenBoard 公共 URL 和技术雷达配置。
- Playwright 使用受控 `/api/about-status` 响应验证中英文内容、热力格、Token 数据、微信读书空状态、技术雷达和移动端无溢出。
- 视觉验收使用构建后的静态产物，检查 `1440 x 1000` 与 `390 x 844`，不启动开发服务器。

## 待扩展项

- 确认可长期使用的微信读书数据来源后，增加独立同步脚本和 CI 调度。
- TokenBoard 公共 API 提供逐日趋势后，可在保持现有摘要的前提下增加时间序列。
- GitHub 页面结构变化时，需要根据受控失败日志更新解析器和夹具。
