## 设计

### 职责

站点公共配置负责集中保存更换博客所有者时需要调整的公开信息，包括品牌、作者、头像、栏目介绍、社交链接以及第三方服务公开 ID。页面、SEO、RSS 和评论系统只读取这份配置，减少散落在组件里的硬编码。

### 目录结构

| 路径                                     | 用途                                |
| ---------------------------------------- | ----------------------------------- |
| `site.config.ts`                         | 唯一的公开配置编辑入口              |
| `lib/site-config.ts`                     | 多语言读取助手和站点 URL 校验       |
| `lib/giscus/config.ts`                   | 评论开关与 Giscus 字段格式校验      |
| `lib/analytics/config.ts`                | Umami 与 Google Analytics 配置校验  |
| `lib/monetization/config.ts`             | 广告、赞赏和本地素材路径校验        |
| `tests/unit/site-config.test.ts`         | 多语言资料、集成开关和站点 URL 单测 |
| `tests/unit/giscus-config.test.ts`       | 评论配置启用、关闭和异常组合单测    |
| `tests/unit/monetization-config.test.ts` | 广告与赞赏配置边界单测              |

### 关键决策

1. `site.config.ts` 放在仓库根目录，作为更换站点资料时首先编辑的文件。
2. 配置按品牌、作者、栏目文案、社交链接和第三方集成分组；中英文值使用同一套语言键。
3. 头像可以填写 `public/` 下的站内路径或公开 HTTPS URL，分享图可以填写站内路径。社交链接可以单独关闭，关闭后页面不会留下空入口。
4. Giscus、AdSense 客户端 ID、Umami Website ID/分享 ID、GA4 Measurement ID、AutoRAG 名称和模型名称都属于公开部署参数，可以保存在公共配置中。
5. Notion Token、未来新增的服务密钥等私密值只存放在环境变量中。`NEXT_PUBLIC_SITE_URL` 由部署环境提供，以便本地、预览和生产使用不同域名。
6. 每项第三方集成都有独立开关。关闭后不会渲染对应入口，也不会发起相关浏览器请求。

## 实现细节

### 配置分组

| 分组                  | 主要字段                                               |
| --------------------- | ------------------------------------------------------ |
| `brand`               | 站名、字标、品牌缩写、图书标识、简介、页脚短句、分享图 |
| `author`              | 作者名、签名、首页标题、关于页文案、头像与替代文字     |
| `sectionDescriptions` | 文章、归档和图书栏目的中英文说明                       |
| `socialLinks`         | 平台、显示名称、URL 和开关                             |
| `integrations`        | Giscus、广告、赞赏、Umami、GA4、Cloudflare AI Search   |

`getLocalizedSiteConfig(locale)` 从多语言配置中取出当前页面需要的值。客户端组件只读取公开信息；服务端元数据和 RSS 也复用同一个结果。

外部头像通过浏览器直接请求，图片服务仍能看到访问者 IP 和 User-Agent。头像请求使用 `no-referrer`，不会附带当前博客页面地址；对隐私要求更高时应把头像放到 `public/images/`。

### 第三方服务

- `integrations.comments.enabled` 已开启文章评论入口；`repo`、`repoId`、`category` 和 `categoryId` 使用 Giscus 生成的真实公开值。
- `integrations.advertising` 管理三个广告位置、AdSense Publisher/slot 和自营广告内容；当前启用文章位置，全部位置关闭时不会加载 Google 脚本。
- `integrations.sponsorship` 管理文章赞赏开关、站内二维码路径和中英文文案。
- `integrations.analytics` 配置 Umami 服务地址、API 路径、时区、Website ID、公开分享 ID、采集和文章浏览量开关。
- `integrations.googleAnalytics` 配置 GA4 Measurement ID 与开关；启用后延后加载外部脚本，站内路由变化也会上报 page view。
- `integrations.aiSearch` 配置公开 API 路径、AutoRAG、模型和每用户/全站限流阈值；开关关闭后不渲染浮动入口，服务端 binding 仍由 `wrangler.jsonc` 管理。
- Playwright 使用 `PLAYWRIGHT_TEST_COMMENTS=1` 注入固定测试评论配置，不读取真实仓库，也不会访问真实 GitHub 数据。

### 更换站点资料

1. 修改 `site.config.ts` 的品牌和作者资料。
2. 将头像与分享图放入 `public/`，更新对应站内路径。
3. 修改或关闭社交链接。
4. 按需启用评论、广告、赞赏、访问统计和 AI 搜索；更换 Umami 账号时同时替换 Website ID、分享 ID、API 路径和时区，更换 GA4 时替换 Measurement ID。
5. 更换 AutoRAG 时同步更新 `integrations.aiSearch`，并确认 Cloudflare 项目中存在对应索引与 AI binding。
6. 修改部署环境中的 `NEXT_PUBLIC_SITE_URL`，再运行完整项目门禁。

## 测试方法

- 单测确认两种语言都能读取完整的站点名称、作者和描述。
- 单测确认当前第三方开关与真实迁移配置一致，并覆盖关闭和非法配置分支。
- Giscus 单测覆盖关闭、完整启用、格式错误和 Playwright 测试配置。
- AI 搜索单测覆盖公开配置被 API 与客户端共同读取，E2E 验证入口、请求与失败状态。
- E2E 继续验证首页标题、作者区、文章评论和英文 RSS 路径。

## 当前限制

- 当前只支持一位站点作者。
- 三个 AdSense 广告位已经实现，文章位置使用旧项目 Publisher/slot 并已开启；首页和列表位置需要在 AdSense 后台建立合适单元后再开启。
- Umami、文章浏览量、GA4、CSP 和静态产物检查已经接通；更换任一服务时需要同步检查浏览器请求域名。
- 微信赞赏区已经实现，真实收款码仍需由站点所有者放入 `public/images/` 并人工确认收款对象。
