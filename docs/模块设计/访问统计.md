## 设计

### 职责

访问统计模块负责延后加载 Umami Cloud 与 Google Analytics，提供旧站公开统计面板入口，并在文章页和文章卡片显示 Umami 浏览量。它不读取正文，不参与内容排序；第三方接口或脚本失败时只隐藏或降级统计信息，不影响阅读、搜索和评论。

### 关键决策

1. 继续使用旧站的 Umami Cloud Website ID、分享 ID 和 GA4 Measurement ID，延续同一站点的数据来源。
2. Umami 和 GA4 在 hydration 后由客户端加载器异步插入。这样既不阻塞正文，也能避开 React 静态导出给服务端脚本标签添加 `inert` 后无法执行的问题；GA4 的本地初始化只负责排队，并默认将分析与广告存储设为 `denied`。
3. Umami 页面统计在浏览器请求，构建不依赖外部 API。分享令牌缓存 24 小时，文章统计缓存 1 小时；同一页面上的重复组件复用进行中的请求。
4. 当前 canonical 和 `redirects.config.ts` 中指向它的旧文章路径会并行查询后求和，使改过 slug 的文章继续显示迁移前数字。英文文章同时查询旧 `/posts/en/<slug>/` 与新 `/en/posts/<slug>/`。
5. Umami Cloud 当前要求统计请求同时带 `x-umami-share-token` 和 `x-umami-share-context: 1`。遇到 `401` 时清理令牌并只重试一次，避免无限重试。
6. 配置启用时要求 HTTPS 服务地址、同源 API 路径、有效 IANA 时区、UUID Website ID、安全分享 ID 和 GA4 `G-` 格式 ID；错误配置会阻止构建。
7. CSP 精确放行 Umami、Google Tag Manager 和 GA4 收集域名，不新增第三方 iframe 或表单来源。

## 实现细节

`resolveUmamiConfig` 和 `resolveGoogleAnalyticsConfig` 是统计配置入口。开关关闭时返回 `null`；开启时生成外部脚本、公开面板与 Umami Share API 地址。`ExternalScript` 在 hydration 后创建带稳定 ID 的异步脚本，同一浏览器会话只插入一次，加载失败时移除标签以便后续重试。`AnalyticsScript` 使用 `data-website-id` 记录页面，GA4 组件关闭自动 page view 后自行发送首次打开和后续站内路径变化，避免重复计数。

`ArticleViewStats` 先读取本地缓存，再取一次公开分享令牌，并用 `Promise.all` 并行查询当前/旧路径。文章页显示浏览和访客，卡片只显示浏览量，给固定最小宽度降低加载后的布局变化。首页和页脚只有在配置有效且存在分享 ID 时显示公开统计链接。

- E2E 拦截 Umami、GA4 和 AdSense 请求，确认浏览器挂载的标签可执行、模拟脚本产生真实页面状态、文章展示合并后的历史数字且页面无控制台错误。

## 当前限制

- Umami Cloud 的统计字段和数据保留策略由 Umami 服务控制；更高的数据控制要求需要改为自托管 Umami。
- 旧路径与当前路径的访客数直接相加，同一读者访问过两个地址时可能重复计入；公开 API 无法跨路径去重。
- 浏览器阻止第三方请求时，文章页显示“统计暂不可用”，卡片也会降级显示该状态，其他内容保持可用。
