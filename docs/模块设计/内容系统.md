# 内容系统

## 设计

### 职责

内容系统负责读取仓库内的文章和图书，检查 frontmatter、目录位置、slug、译文关联和图片路径，再生成页面可直接使用的正文信息、阅读数据、标签统计和归档分组。它只处理构建期数据，不负责正文渲染、分页界面、搜索交互和在线内容管理。

### 目录结构

| 路径                           | 用途                                   |
| ------------------------------ | -------------------------------------- |
| `content/posts/{zh,en}/`       | 中英文文章源文件                       |
| `content/books/{zh,en}/`       | 中英文图书源文件                       |
| `lib/content/schema.ts`        | 文章和图书 frontmatter 的 Zod 严格结构 |
| `lib/content/reader.ts`        | 文件发现、读取、校验和数据组装         |
| `lib/content/markdown.ts`      | Markdown 纯文本和二、三级目录提取      |
| `lib/content/metrics.ts`       | 中英文字数和阅读时间计算               |
| `lib/content/queries.ts`       | 发布过滤、排序、归档、标签和译文查询   |
| `lib/content/types.ts`         | 页面层消费的只读内容类型               |
| `scripts/validate-content.ts`  | 命令行内容检查入口                     |
| `tests/unit/content-*.test.ts` | 内容规则单元测试                       |

### 关键决策

1. 内容保存在仓库内，构建时一次性读取。文章、标签、归档和图书页面因此可以完整静态导出。
2. 文章和图书都使用 Markdown/MDX 文件。frontmatter 只接受 schema 声明的字段，拼错字段会立即失败。
3. 文件名就是 slug，必须使用小写 kebab-case。中文内容放在 `zh/`，英文内容放在 `en/`，目录与 `locale` 不一致会失败。
4. `translationKey` 只在确实存在译文时填写。同一个 key 必须连接两种语言，并且每种语言只能出现一次。
5. 图书正文中的二级、三级标题就是章节目录。文章使用同一套标题分析生成文内目录。
6. 内容文件并行读取；文章和图书也在校验脚本中并行加载，避免构建阶段串行等待。
7. 页面层只使用 `lib/content` 返回的数据。草稿过滤、置顶、日期排序和译文查找不在页面里重复实现。

## 实现细节

### 读取流程

1. `loadArticles` 或 `loadBooks` 递归列出 `.md`、`.mdx` 文件并按路径排序。
2. `gray-matter` 拆分 frontmatter 和正文，Zod 检查字段类型、必填项、日期范围、标签重复和进度范围。
3. reader 检查文件名 slug、`locale` 所在目录、译文 key 唯一性和文章图片是否存在。
4. Markdown AST 提取纯文本与二、三级标题。相同标题使用稳定且唯一的锚点。
5. 文章纯文本按每分钟 300 个中文字符、200 个英文单词估算阅读时间，最少显示 1 分钟。
6. reader 返回整理后的只读对象；查询函数再按语言和草稿状态生成发布清单。

### 排序与查询

- 文章先按 `pinned` 置顶，再按 `published` 从新到旧排序。
- 图书先排除草稿并按语言筛选，再按 `order` 从小到大排序。
- 年份归档沿用文章顺序，标签统计按数量降序、名称升序排列。
- 译文只通过相同 `translationKey` 和目标语言匹配，并排除草稿，缺少译文时返回 `null`。
- 列表和首页使用 `ArticlePreview`，不会把正文、纯文本、目录和源文件路径传给客户端组件。
- `plainText` 与标题、摘要和标签一起写入 MiniSearch 索引；正文只存在于构建期索引输入，不会作为页面属性传给客户端组件。

### 异常处理

所有可预期的内容问题都抛出 `ContentValidationError`，消息以源文件路径开头，再说明具体字段或规则。`pnpm content:check` 捕获错误后返回非零退出码，`pnpm build` 因此会停止。

### 测试方法

- reader 测试在系统临时目录生成空内容、单篇内容和异常内容，覆盖日期、图片、译文、重复 slug 与语言目录。
- 纯函数测试覆盖目录锚点、纯文本、字数、阅读时间、发布筛选、置顶排序、归档、标签、图书顺序和译文映射。
- 完整 `pnpm content:check` 校验仓库中的全部真实内容和图片引用。

## 当前限制

- 当前正文只渲染安全 Markdown，不执行原始 HTML 或任意 MDX React 组件。
- 作者资料来自 `lib/site-config.ts`，首版只支持一位站点作者。
- Notion 仍未成为在线内容源；后续同步会先落成本地 Markdown，再经过同一套校验和构建流程。
