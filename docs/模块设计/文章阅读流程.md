## 设计

### 职责

文章阅读流程负责把内容系统生成的数据呈现为首页近期文章、文章列表和独立正文页面，并完成标签筛选、标签详情入口、分页、目录锚点、前后篇跳转、广告、赞赏和评论入口。归档展示、搜索索引和完整 SEO 由对应模块实现。

### 关键决策

1. 首页、列表和正文都是 Server Component 的入口。只有需要读取浏览器 URL 和响应按钮操作的 `ArticleBrowser` 是 Client Component。
2. 客户端列表只接收 `ArticlePreview`，不包含正文、搜索纯文本、目录和源路径，减少页面脚本数据。
3. 标签筛选使用 HeroUI Button，文章摘要使用 Card 和 Chip，空结果与分页使用对应 HeroUI 组件。项目样式只调整尺寸、颜色和排版。
4. 筛选与页码保存在 `?tag=`、`?page=` 中。组件通过浏览器历史记录同步状态，支持刷新和后退，也不会触发额外的服务端数据请求。
5. 每种语言的 `[slug]` 都由 `generateStaticParams` 在构建时列出，`dynamicParams = false` 防止静态部署生成未知路径。
6. Markdown 使用 `react-markdown` 渲染，不开启原始 HTML。`rehype-slug` 与内容分析器共用 GitHub slug 规则，目录链接和正文标题保持一致。
7. 文章始终在当前标签页进入独立地址。刷新、复制链接和浏览器前进后退都能回到同一篇文章。
8. 正文结束后依次展示正文广告、前后篇、赞赏和评论。评论接近视口后自动加载，第三方服务失败也不会影响前面的阅读流程。
9. 目录不超过 20 项时展示二、三级标题；超过 20 项时只保留二级章节，并限制桌面目录高度，避免长文目录超过视口。
10. 首页近期文章和列表文章卡片都使用整卡站内链接。封面是可选区域：有图时使用横向比例，无图时正文占满；标题和摘要都限制行数，保证同一行卡片等高。

## 实现细节

### 首页与列表

1. `HomePage` 和 `PostsPage` 通过缓存后的 `getAllArticles` 读取文章。
2. 内容查询先排除草稿、筛选语言，再按置顶和日期排序。
3. 首页取前 3 篇文章和前 6 个标签；近期文章整卡直接指向对应的独立文章地址，并保留可见的“阅读全文”作为操作提示。
4. 列表把轻量摘要传给 `ArticleBrowser`。浏览器根据 URL 计算当前标签、页码和可见文章，不另存一份容易不同步的派生状态。
5. 用户选择标签或页码时写入浏览器历史并触发重新计算。无效标签回到全部，无效页码限制在实际页数内。
6. `ArticleCard` 用一个 `SiteLink` 包住 HeroUI Card。卡片内部只保留文字和 Chip，没有第二个链接；键盘焦点、悬停和任意空白区域点击都进入同一文章。
7. 有封面时渲染 `16:10` 媒体区，无封面时不输出媒体节点。桌面和手机分别使用固定卡片与网格行高，分页预留空间只留在列表底部；摘要通过两行截断防止内容长度改变布局。文章列表不再渲染重复的“阅读全文”，标签在桌面和手机端都通过自动外边距固定在底部；手机端日期和阅读时长保持单行，正文区不能撑出卡片边界。

### 正文

1. 路由根据语言和 slug 查找已发布文章，找不到时进入站点 404。
2. 页头显示发布日期、字数、阅读时间、摘要和标签；标签进入当前语言的独立详情页。
3. 正文由服务端生成 HTML。迁移内容只保留页头文章标题为一级标题，正文二、三级标题带稳定 id，目录链接滚动到对应位置。
4. 桌面目录固定在正文右侧并限制在当前视口内；手机端移到正文上方。长文只列二级章节，短文继续显示二、三级标题。
5. 前后篇沿用文章列表顺序。置顶文章优先，其余文章按发布日期从新到旧排列。
6. 广告和赞赏配置由服务端校验后传给展示组件。正文广告位于 Markdown 与前后篇之间，赞赏位于前后篇与评论之间。
7. 评论配置由服务端从 `site.config.ts` 读取，客户端只接收公开的仓库与分类 ID。评论区接近视口后插入 `giscus.app/client.js`，iframe 加载完成后主题变化通过 Giscus 消息协议同步。

## 当前限制

- 当前正文支持安全 Markdown 语法，不执行 MDX 中的任意 React 组件或原始 HTML。
- Giscus 登录、发言和反应依赖 GitHub 服务；第三方不可用时文章正文仍可正常阅读。
