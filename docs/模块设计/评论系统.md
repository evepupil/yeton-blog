# 评论系统

## 设计

### 职责

评论系统负责在中英文文章末尾提供 Giscus 讨论入口，处理公开配置校验、按需加载、明暗主题联动、超时、脚本失败和重试。正文、前后篇和站内导航不依赖 Giscus，评论服务不可用时文章仍可完整阅读。

### 目录结构

| 路径                                    | 用途                            |
| --------------------------------------- | ------------------------------- |
| `lib/giscus/config.ts`                  | 四项公开配置的完整性和格式校验  |
| `features/comments/giscus-comments.tsx` | 客户端脚本加载、状态和主题消息  |
| `features/posts/article-page.tsx`       | 将评论区接到中英文文章阅读流    |
| `features/posts/posts.css`              | 评论区桌面和手机样式            |
| `public/_headers`                       | 放行 Giscus 脚本、连接和 iframe |
| `tests/unit/giscus-config.test.ts`      | 配置缺失、完整和异常组合测试    |
| `tests/e2e/home.spec.ts`                | 评论加载、主题联动和失败降级    |

### 关键决策

1. Giscus 使用 `pathname` 映射和严格模式。中文与英文文章路径不同，因此各自拥有独立讨论。
2. 评论采用明确的“加载评论”按钮。点击前页面不会连接 `giscus.app`，减少首屏资源和无意的第三方请求。
3. 仓库、仓库 ID、分类和分类 ID 都是公开值，通过 `NEXT_PUBLIC_GISCUS_*` 在构建期传入。四项全空时关闭评论，部分配置会让构建失败。
4. `next-themes` 提供当前明暗主题。iframe 首次创建时使用当前主题，后续切换通过 `{ giscus: { setConfig: { theme } } }` 消息更新。
5. 脚本错误或 15 秒内没有可用 iframe 时进入失败状态。失败状态保留文章内容，并提供重新加载按钮。
6. CSP 只为 `https://giscus.app` 增加脚本、连接和 iframe权限，没有扩大图片、表单或其他第三方来源。

## 实现细节

### 公开配置

| 变量                             | 当前值或来源                                 |
| -------------------------------- | -------------------------------------------- |
| `NEXT_PUBLIC_GISCUS_REPO`        | `evepupil/yeton-blog`                        |
| `NEXT_PUBLIC_GISCUS_REPO_ID`     | `R_kgDOTY-rvQ`，由 GitHub 公开 REST API 确认 |
| `NEXT_PUBLIC_GISCUS_CATEGORY`    | 仓库启用 Discussions 后在 Giscus 页面选择    |
| `NEXT_PUBLIC_GISCUS_CATEGORY_ID` | Giscus 根据所选 Discussion 分类生成          |

配置值会出现在页面和网络请求中，不应当存放 Token、密码或其他私密数据。

### GitHub 前置设置

1. 使用仓库所有者账号进入 `evepupil/yeton-blog -> Settings -> General -> Features`，开启 Discussions。
2. 安装 Giscus GitHub App，并只授权需要评论的仓库。
3. 打开 `giscus.app`，输入仓库，选择一个允许新讨论的分类，复制分类名称和 ID。
4. 将四项配置加入 Cloudflare Pages 的生产与预览构建环境；确认构建日志中已经注入。
5. 部署后打开一篇文章，加载评论并完成一次测试讨论，再验证明暗主题和登录流程。

### 客户端状态

- `disabled`：四项配置都缺少，显示评论暂未开放。
- `idle`：配置完整，等待读者点击加载。
- `loading`：脚本已插入，等待 Giscus iframe 完成加载。
- `ready`：iframe 可用，主题变化通过消息同步。
- `error`：脚本失败或超时，显示说明和重试按钮。

组件卸载或文章切换时会清理超时器、DOM 观察器、脚本和 iframe，避免把上一页讨论带到下一篇文章。

### 隐私边界

点击加载前只渲染本站 HTML、CSS 和 JavaScript。点击后浏览器会连接 Giscus 和 GitHub，第三方可能接收 IP、User-Agent、GitHub 登录状态以及评论交互数据。本站不代理、不复制评论内容，也不额外保存 GitHub 身份信息。

### 测试方法

- 配置单测覆盖全空、完整、部分缺失和仓库格式错误。
- Playwright 拦截 `giscus.app/client.js`，用同源测试 iframe 接收主题消息，不依赖真实 GitHub。
- 失败测试在 `390 x 844` 视口中断脚本请求，确认错误提示和重试按钮出现，同时文章标题仍可见。
- `pnpm build` 检查 `_headers` 包含 Giscus 所需的 `script-src` 与 `frame-src`。

## 当前限制
